# Grafyx 0.2 — Senior-Engineer Substrate for LLMs

**Date:** 2026-04-30
**Owner:** Bilal Karadeniz <bilal07karadeniz@gmail.com>
**Target version:** `grafyx-mcp` 0.2.0 on PyPI
**Status:** Design — pending user approval before plan

## Goal

Make Grafyx the only MCP that gives an LLM a senior engineer's structural awareness of the user's specific repo — duplicates, hotspots, architecture, co-edits, semantic neighborhoods — through tools that are **either deterministic (100%) or matched to published state-of-the-art** for their task.

The pitch: *"Grafyx is the only MCP that gives an LLM a senior engineer's mental model of the user's repo, learned from the repo itself, without a daemon, without a GPU, without sending code to a server."*

## Non-goals

- Generating architecture suggestions, narrative summaries, or "smart" recommendations. The LLM does synthesis. Grafyx provides verifiable substrate.
- Training new from-scratch encoders or cross-encoders. M5/M6 directions are closed permanently.
- Natural-language inputs to tools (other than for retrieval ranking inside `find_related_code`).
- Daemon, GPU dependency, cloud calls. Stays `pip install grafyx-mcp` + numpy/ONNX-CPU.
- Fixing the Celery `.delay()` dynamic dispatch and TS object-literal counting limitations — separate bug-fix track, post-0.2.

## Hard discipline (non-negotiable)

1. **Concrete in, concrete out.** Inputs: symbol names, file paths, diffs, integer thresholds. Outputs: structured records with numeric scores. No prose generation.
2. **Every tool has a benchmark.** If we can't write a measurable eval, we don't ship the tool.
3. **No homegrown ML that loses to off-the-shelf.** Use `fastembed` ONNX encoders, `datasketch` MinHash, `networkx` graph algorithms. Build new only where nothing exists (clone detection tuned for MCP, co-edit graphs over MCP, hotspot ranking).
4. **Existing M1/M3/M4/gibberish detector stay.** They're working post-processing layers.
5. **M5 retires.** Replaced by pretrained ONNX encoder.

## Section 1 — Tool surface for v0.2

Six tools shipped in v0.2: 4 new deterministic tools, 1 new approximate tool, 1 rebuilt tool, plus 1 small primitive that shares the encoder.

### 1.1 `get_hotspots` — *deterministic, 100% by construction*

```
get_hotspots(
  top_n: int = 20,
  since: str = "1 year",
  path_prefix: str = "",
) -> list[{
  file: str,
  churn_count: int,
  lines_changed: int,
  fan_in: int,
  age_days: int,
  score: float,
}]
```

**Score formula (locked):**
```
score = log2(churn_count + 1) * (fan_in + 1) * exp(-age_days_since_last_edit / 365)
```

### 1.2 `get_architecture_metrics` — *deterministic*

```
get_architecture_metrics(
  module_path: str = "",
) -> {
  modules: list[{
    path: str,
    file_count: int,
    fan_in: int,
    fan_out: int,
    internal_density: float,
    betweenness: float,
    depth: int,
    cycles: int,
  }],
  global: { total_modules, isolated_modules, scc_count, longest_cycle }
}
```

No classification of modules into "utility" / "core" / etc. Just numbers. The LLM decides.

### 1.3 `get_co_edited_files` — *deterministic*

```
get_co_edited_files(
  file: str,
  min_co_edits: int = 3,
  since: str = "2 years",
  top_n: int = 10,
) -> list[{
  file: str,
  co_edit_count: int,
  co_edit_ratio: float,         # co_edits(A,B) / max(total_edits(A), total_edits(B))
  last_co_edit: str,            # ISO date
}]
```

Historical fact, not prediction. Output is a frequency table.

### 1.4 `find_duplicate_functions` — *≥92% F1 on Type-1/2/3*

```
find_duplicate_functions(
  min_similarity: float = 0.85,
  min_lines: int = 5,
  path_prefix: str = "",
  match_types: list[str] = ["type1", "type2", "type3"],  # opt-in to type4
) -> list[{
  pair: [
    { symbol, file, line_start, line_end },
    { symbol, file, line_start, line_end },
  ],
  similarity: float,
  match_type: "type1"|"type2"|"type3"|"type4",
  evidence: { ast_jaccard: float, encoder_cosine: float? },
}]
```

### 1.5 `find_related_code` — *rebuilt, candidate ranker, SOTA encoder*

```
find_related_code(
  query: str,
  k: int = 10,
  min_similarity: float = 0.0,
  scope: "functions"|"classes"|"both" = "both",
) -> {
  model: { id: str, version: str },
  results: list[{
    symbol, file, line_range, kind,
    similarity_score: float,
    match_reason: "embedding"|"name"|"source",
  }]
}
```

Output explicitly includes similarity scores so the LLM (and user) see this is a ranked list, not a verdict.

### 1.6 `get_similar_functions` — *small primitive, same encoder*

```
get_similar_functions(
  symbol: str,
  k: int = 10,
  min_similarity: float = 0.5,
  exclude_same_file: bool = False,
) -> list[{ symbol, file, line_range, similarity_score }]
```

Anchored on a symbol instead of a query. Reuses the encoder index. ~15 LOC wrapper.

## Section 2 — Encoder swap (M5 → CodeRankEmbed)

### 2.1 License audit eliminated the obvious "best" picks

| Candidate | Params | License | CoIR nDCG@10 | Verdict |
|---|---|---|---|---|
| jina-code-embeddings-0.5b | 500M | CC-BY-NC-4.0 | SOTA | ❌ non-commercial |
| Salesforce SFR-Embedding-Code-400M_R | 400M | CC-BY-NC-4.0 | 61.9 | ❌ non-commercial |
| Nomic Embed Code 7B | 7B | Apache 2.0 | top-tier | ❌ too big for CPU-first |
| CodeSage Large V2 | 1.3B | Apache 2.0 | strong | ❌ borderline-too-big |
| **CodeRankEmbed (nomic-ai)** | **137M** | **MIT** | **60.1** | ✅ **primary pick** |
| jina-embeddings-v2-base-code | 137M | Apache 2.0 | older gen | ✅ fallback |
| Qwen3-Embedding-0.6B | 600M | Apache 2.0 | strong on MTEB Code | ⚠️ deferred for evaluation |

### 2.2 Primary: `nomic-ai/CodeRankEmbed`

- **MIT licensed** — commercial-permissive, no asterisks
- **137M params** — same CPU class as fallback, ~30–50 ms p50
- **CoIR nDCG@10 = 60.1** — only 1.8 points behind the best non-commercial small model
- **8192-token context** — handles long functions and full file embeddings
- **MRR 77.9 on CodeSearchNet**
- Base: `snowflake-arctic-embed-m-long`
- Requires query prefix `"Represent this query for searching relevant code: ..."` — handled inside `find_related_code`

### 2.3 Distribution path

- Convert weights to ONNX-int8 once at release time (~50–80 MB quantized)
- Host ONNX bundle on HuggingFace under `bilal07karadeniz/grafyx-coderankembed-onnx`
- Plug into `fastembed` via `add_custom_model()`
- Runtime deps: `fastembed` + `onnxruntime` only (no `torch`, no `sentence-transformers`)
- First-run download cached at `~/.cache/fastembed/`
- Total install footprint after first run: ~80 MB encoder + ~15 MB onnxruntime

### 2.4 Fallback: `jinaai/jina-embeddings-v2-base-code`

In `fastembed`'s official default model list. Apache 2.0. Zero packaging work. Lower CoIR score. Used if CodeRankEmbed conversion fails or its real-world delta on benchmark repos is <3 nDCG@10 vs jina-v2 (small-effect threshold).

### 2.5 Index lifecycle

- Built async at startup, after graph parse (non-blocking for deterministic tools)
- Format: `{symbol_id: float16[768]}`, ~30 MB per 10k functions on disk
- Watcher integration: re-embed only changed symbols on file save (existing debounce)
- Rebuild on encoder version change

### 2.6 Layering with existing models

- M5 retires. Delete `_code_encoder.py`, BPE assets (`bpe_*.json`, `code_encoder_weights.npz`), `_mamba.py` `FeedForwardBlock` if not used elsewhere.
- M1 stays as **re-rank layer** on encoder's top-50 candidates. M1's structural features compose with the encoder.
- M3 source filter, M4 importance, gibberish detector, M2 caller disambig: all unchanged.

### 2.7 Failure modes

- Encoder download fails → encoder-using tools return `{ok: false, reason: "encoder_unavailable"}`. Deterministic tools keep working. `find_related_code` falls back to source-token search with `degraded: true`.
- Slow user CPU → response includes latency in metadata.

### 2.8 Performance commitments

- Index build: ≤10 min for 10k symbols on modern CPU
- `find_related_code` p50 latency: ≤100 ms; p95 ≤200 ms
- Peak RAM overhead: ≤500 MB
- Wheel size: smaller than 0.1.1 (M5 deletion offsets)

## Section 3 — Clone detection algorithm (`find_duplicate_functions`)

### 3.1 Three-stage pipeline

**Stage 1 — Candidate generation (MinHash LSH).** For each function, build AST-token shingles (length 5–7 sliding window over normalized AST node types + identifiers). Hash with MinHash (`datasketch`, 128 permutations). LSH bucket at target Jaccard ≈ `min_similarity`. O(n) candidate generation with high recall.

**Stage 2 — Deterministic verification.** For each candidate pair:
- AST-with-identifiers Jaccard → if ≥0.95, label `type1`
- Identifier-stripped AST Jaccard → if ≥0.90, label `type2`
- Tree edit distance ratio (Zhang-Shasha or `zss`) → if ≥`min_similarity`, label `type3`

**Stage 3 — Type-4 (semantic) opt-in.** Only if `match_types` includes `"type4"`. Encoder cosine ≥0.88 → label `type4`. Output includes both `ast_jaccard` and `encoder_cosine` so the LLM sees this is low-evidence semantic guess, not a structural clone.

### 3.2 Exclusions (precision protection)

- Functions <`min_lines` lines (default 5)
- Test functions: `test_*`, `*_test`, `it()`, `describe()` blocks
- Convention boilerplate: `__init__`, `__repr__`, `__str__`, `__eq__`
- Generated files: `*_pb2.py`, `*.generated.*`, configurable path filter

### 3.3 Performance budget

- 10k functions: <30 s for stages 1+2
- Stage 3: ~5 ms/pair, only on opt-in
- Memory: <300 MB peak

### 3.4 Library choices

- `datasketch` — Apache 2.0, MinHash + LSH, mature
- AST extraction reuses `graph-sitter` — no new dep
- Tree edit distance: `zss` (MIT, ~200 LOC) or hand-rolled Zhang-Shasha

## Section 4 — Git mining (`get_hotspots`, `get_co_edited_files`)

### 4.1 Library choice: `subprocess` to `git`

Rejected alternatives: `pygit2` (heavy libgit2 system dep), `GitPython` (slow on big repos). Every Grafyx user has `git` installed. `git log --numstat --pretty=format:"%H|%ct"` on a 100k-commit repo finishes in ~3–5 s. No new Python deps.

### 4.2 Index structure

In-memory:
- `_git_index: {file_path: {commits: [hash], churn: int, lines_changed: int, first_seen_ts: int, last_modified_ts: int}}`
- `_co_edit_index: {file_path: {co_file_path: count}}` — sparse, stored only when count ≥ 2

### 4.3 Build & refresh

- Lazy: built on first call to a git-using tool
- Persisted to `~/.cache/grafyx/<project_path_hash>/git_index.json`
- Background refresh: poll `git rev-parse HEAD` every 30 s; on change, run `git log <cached_head>..HEAD --numstat` and merge incrementally

### 4.4 Edge cases

- Not a git repo → `{ok: false, reason: "not_a_git_repo"}`
- Shallow clone → warn, work with available history
- Submodules → skip
- Renamed files → opt-in `follow_renames=False` default (perf cost of `--follow`)

### 4.5 Hotspot formula (locked)

```
score = log2(churn_count + 1) * (fan_in + 1) * exp(-age_days_since_last_edit / 365)
```

### 4.6 Co-edit ratio (locked)

```
co_edit_ratio = co_edits(A, B) / max(total_edits(A), total_edits(B))
```

## Section 5 — Architecture metrics (`get_architecture_metrics`)

### 5.1 Library: `networkx`

BSD licensed, mature. **Opt-in extra**: `pip install grafyx-mcp[graph]`. Falls back gracefully if missing — tool returns `{ok: false, reason: "networkx_not_installed", action_hint: "pip install grafyx-mcp[graph]"}`.

### 5.2 Module definition

- Python: directory containing `__init__.py`, or top-level `.py` files at depth=1 grouped by parent directory
- TS/JS: directory at depth=1 from src root

### 5.3 Algorithms

- `fan_in`, `fan_out`: from existing import index
- `internal_density = internal_imports / (internal_imports + fan_out)`
- `betweenness`: `networkx.betweenness_centrality()`
- `depth`: longest dependency chain ending at module (DAG longest path; condense SCCs first)
- `cycles`: `networkx.simple_cycles()`, capped at 100

### 5.4 Performance

Betweenness is O(VE). Project graphs typically ≤2000 module nodes. Cap at 5000 modules; over that, return per-module on-demand.

## Section 6 — Benchmark harness (`benchmarks/`, gitignored)

### 6.1 Layout

```
benchmarks/
  .gitignore                # whitelist: ignores everything except scripts/ and README.md
  README.md
  repos/                    # cloned at pinned commits
    fastapi/
    django/
    home-assistant/
  eval_data/
    clones/                 # injected clone pairs per repo
    queries/                # NL→symbol pairs (CodeSearchNet Python slice)
    co_edits/               # held-out commit slices
  scripts/
    setup_repos.py          # clones repos at pinned hashes
    inject_clones.py        # generates type-1/2/3/4 clones from real functions
    bench_clones.py
    bench_search.py         # head-to-head: M5 vs jina-v2 vs CodeRankEmbed
    bench_co_edits.py
    bench_metrics.py
    run_all.py              # entry point
  results/
    YYYY-MM-DD/<run_id>/
      clones.json
      search.json
      co_edits.json
      summary.md
```

### 6.2 `benchmarks/.gitignore` (whitelist pattern)

```
*
!.gitignore
!README.md
!scripts/
!scripts/**
```

Scripts version-controlled; repos, eval data, results are not.

### 6.3 Reproducibility

Every result JSON includes:
```json
{
  "repo": "fastapi",
  "commit": "<sha>",
  "grafyx_sha": "<sha>",
  "encoder": "CodeRankEmbed@v1",
  "hyperparams": { "min_similarity": 0.85, "..." },
  "run_started_ts": "...",
  "run_finished_ts": "..."
}
```

### 6.4 Execution

`cd benchmarks && python -m scripts.run_all` — single command, single JSON+markdown summary. No CI integration in v0.2.

### 6.5 Eval methodology per tool

- **Clone detection:** inject 50 clones per repo (10 each: type-1, type-2, type-3, type-4, hard-negative pairs). Sweep threshold. Measure precision/recall/F1 per type.
- **`find_related_code`:** existing M5 eval set + CodeSearchNet Python slice. Report nDCG@10, MRR@10. Head-to-head M5 vs jina-v2 vs CodeRankEmbed.
- **Co-edit prediction:** hold out last 20% of commits per repo. Predict co-edited files for each held-out commit's first changed file from earlier history. Measure precision@5.
- **Hotspots:** smoke test only — verify formula matches expectation on fixture.
- **Architecture metrics:** unit tests on synthetic graphs with known properties.
- **`get_similar_functions`:** inherits the encoder quality metrics from `find_related_code` (same encoder index). Sanity-checked with 30 hand-labeled anchor pairs per repo, target precision@5 ≥ 0.80.

## Section 8 — Risks & mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| CodeRankEmbed ONNX conversion fails | medium | Fallback jina-v2-base-code (in fastembed default), benchmark still proceeds |
| Clone detection <92% F1 on Type-3 | medium | Tighten threshold, ship Type-1/2 only with Type-3 behind opt-in flag |
| Betweenness slow on giant repos | low | Cap at 5k modules, on-demand per-module above cap |
| Git history shallow/absent | low | Graceful empty results with `degraded: true` and reason |
| `fastembed` API breaks on minor version | low | Version-pin in `pyproject.toml` |
| User CPU too slow for encoder | low | Per-query latency in response metadata; document min specs |
| Wheel size grows unexpectedly | low | M5 deletion budget; verify with `python -m build` before release |
